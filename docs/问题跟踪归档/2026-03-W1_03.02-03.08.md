### #001 多语言页面问题

- **提出时间**: 2026年3月2日
- **提出人**: 用户
- **优先级**: P1
- **状态**: 🟢 已解决
- **问题描述**:
  - 页面的整体UI样式不好看，整个页面字体偏小，设计样式不精致（先调用sketch进行设计样式的查看，然后根据的专业建议进行页面UI的优化）
  - 关于英文的切换，整体系统还有多个页面并不会切换成英文，比如「我的-通知」页面、活动详情的所有页面等等，你先仔细进行检查
  - 「我的-通知-@我的」、「报名-活动详情」页面的顶部4个tab栏目对应的每个页面、活动评论的所有页面都没有切换成英文


#### 🔍 分析结果 (2026年3月2日)

##### 子问题一：语言设置页 UI 对比 Sketch 设计稿

**已查看设计稿**: `小程序端设计/我的-设置-多语言选择.png`

| 对比项 | Sketch 设计稿 | 当前实现 | 差距 |
|--------|-------------|---------|------|
| 页面标题 | "语言"，大字体居中 | "语言设置"，18px | 标题文案和大小不一致 |
| 分区标签 | "支持语言"/"其他语言"，清晰可读 | "当前语言"，**10px 极小** | 🔴 字号差距极大 |
| 语言选项字体 | 大字体（约 16-17px），宽松间距 | **12px**，间距紧凑 | 🔴 字号偏小，行高不足 |
| 布局结构 | 两个卡片分区（支持语言 + 其他语言） | 单卡片，仅2项 | 结构不同（可接受，因只支持2种语言） |
| 行高/间距 | 每行约 56px 高，宽松舒适 | min-height 24px + padding 12px | 🔴 过于紧凑 |
| 分割线 | 细线，左右有边距 | 细线，样式一致 | ✅ 基本一致 |
| 返回按钮 | 圆形边框 + 箭头 | 圆形边框 + 箭头 | ✅ 一致 |

**UI 优化方案**：
1. 分区标签字号：10px → **14px**
2. 语言选项字号：12px → **17px**
3. 行最小高度：24px → **52px**
4. 行内间距：padding 12px 0 → **padding 18px 0**
5. 卡片内间距：18px → **24px**
6. 选中指示器尺寸：20px → **24px**
7. 页面标题对齐方式调整为居中

##### 子问题二：i18n 翻译覆盖审计结果

**审计范围**: 所有 pages/ 和 components/ 下的 .tsx 文件

| 页面/组件 | i18n 状态 | 问题描述 |
|-----------|----------|---------|
| `activity-detail/components/OverviewTab.tsx` | 🔴 **完全未接入 i18n** | ~15处硬编码中文：'免费'、'截止报名'、'目前报名人数'、'火热报名中'、'活动介绍'等 |
| `activity-detail/components/HotelTab.tsx` | 🔴 **完全未接入 i18n** | ~10处硬编码中文：设施名称('免费WiFi'、'餐厅'等)、'暂无酒店信息'、'/晚' |
| `credential/index.tsx` | 🟡 有 i18n 但有硬编码默认值 | 默认标题'2025暑期老客户培训会议'、'加载中...'、城市名映射表 |
| `checkin/index.tsx` | 🟡 有 i18n 但有硬编码默认值 | 默认活动标题'暑期培训会议' |
| `comment/index.tsx` | 🟡 有 i18n 但有硬编码默认值 | '用户'（作为兜底名称） |
| 其他页面（index/profile/activities/my-payments/invoice-headers/ai-assistant/language/dark-mode/search） | ✅ 正常 | 已正确使用 useTranslation + t() |

**需新增翻译 key 数量**: 约 20-25 个

**请用户回复**：
1. **UI 优化方案**：上述语言设置页的字号/间距调整方案是否同意？还是有其他要求？
2. **i18n 优先级**：建议按以下顺序修复，是否同意？
   - 第一批（🔴 关键）：OverviewTab.tsx + HotelTab.tsx（活动详情核心页面）
   - 第二批（🟡 次要）：credential + checkin + comment 的硬编码默认值
3. **设计稿中有多种语言选项**（英文UK、英文、中文繁体、印尼语、法语等），但当前只实现了「简体中文」和「English」两种。是否需要扩展更多语言支持，还是保持现状只支持 2 种？

**回复**：

1、认同你的UI优化方案（还有你可以调用sketch的mcp直接查看精准的设计参数）
2、保持现状，支持2种就行
3、我发现「我的缴费」「我的发票抬头」页面的mock数据没有显示，走的应该是真实API的调用，我需要在开发环境下也需要显示mock数据，但是需要保留真实API调用逻辑（详细规范文档可以查看开发流程文档的「真实环境对接」）

### #002 缴费/发票抬头页面 Mock 数据不显示

- **提出时间**: 2026年3月2日
- **提出人**: 用户
- **优先级**: P1
- **状态**: 🟢 已解决
- **问题描述**:
  - 「我的缴费」「我的发票抬头」页面在开发环境下没有显示 mock 数据，走的是真实 API 调用
  - 需要在开发环境下显示 mock 数据，但保留真实 API 调用逻辑
  - 参考：开发流程文档「真实环境对接」章节

#### 🔧 解决方案 (2026年3月2日)

**根因**: `services/payments.ts` 和 `services/invoice-headers.ts` 未检查 `CONFIG.USE_MOCK`，直接调用真实 API；且不存在对应的 mock 数据文件。

**修复内容**:

1. 新增 `mock/payments.ts` — 6 条缴费 mock 数据（含已缴费/未缴费状态，多种分类）
2. 新增 `mock/invoice-headers.ts` — 4 条发票抬头 mock 数据（含个人/单位类型），支持 CRUD 操作
3. 更新 `mock/index.ts` — 导出新增 mock 模块
4. 改造 `services/payments.ts` — 所有函数增加 `CONFIG.USE_MOCK` 判断，mock 模式下返回本地数据
5. 改造 `services/invoice-headers.ts` — 所有函数增加 `CONFIG.USE_MOCK` 判断，mock 模式下支持完整 CRUD
